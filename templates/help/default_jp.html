Gitshellについて

www.gitshell.comその後、技術志向の社会、2012年3月9日で、ドメイン名の登録、2012年3月17日午後09時17分30秒で書かれたコードの最初の行、コードを追加し、週の毎日であることは第6回です。ほとんどの主要な開発時間。開発者として、私はオープンソース、オープンソースの知識を徐々に学習からインストールredhat9.0スタート（私はまだ興奮を覚えている）、からなしグラフィックドライバについて非常に情熱的で、私はオープンソースの世界をお返ししたいと思っています。我々は他の人に有用な製品を作ることであり、gitshellはそのうちの一つであると思いますので、プログラマは、偉大なタイトルでなければなりません。説明最初に、説明書のすべての種類が使用できない回避するために、gitshellは、Google Chromeの、Mozilla Firefoxの、AppleのSafari、オペラや他の近代的なブラウザをサポートしています。

ビジョン

安全で、安定したコミュニティを提供するために、卓越した開発者向けgitshellは、ここでのショーからコードを取得するアイデアを共有し、お互いから学び、開発のレベルを向上させます。

よくある質問

gitshell何？
gitshellクローンを提供し、githubのと同様のGitバージョン管理システムを使用して、1つ、（言語と速度はあなたのために何の問題がない場合、それはgithubのを使用することをお勧めします）ですが、プッシュ、プル、支店管理。 "シェル"を意味し、このサイトでgitshellとのgit - シェルコマンドのように、Linuxのシェルは名前の由来である"シェル"のgitのを提供することです。
なぜgitshell？
プログラマーとして、1日、私は突然、そのコードは、必ずしも多くの価値を持っていませんが、常に失われた変更の様に多くのコードを書くために使用され、実現したが、作家のように、一人一人を残してすることは原稿がある意味のあるもの。 githubの、ビットバケット·プロセスを使用する場合は、速度が（特に一度githubのは、もちろん、githubのではない独自の理由を訪れていない）理想的ではないと感じているので、ローカル環境で構築されたgitのを理解する、自分でgitリポジトリを使用して1つを構築したかったゆっくり考え、なぜ公共gitのホスティング、サービス開発者なので、gitshellた。
gitshellはできますか？
gitのホスティング上gitshellフォーカスは、独創性を奨励するために、特に、軽快なオープンソースコードを歓迎します。良好な開発者コミュニティを作成し、コードは、引数を獲得すること、KISSの原則を信じている。私はもっと面白いことを楽しみにしています。
将来的にはgitshell？
私はgitshellが長い浮かんことを願って、ステップによって、堅牢でふっくらアップステップ。
gitshellアーキテクチャ？
ここhaproxyの、のopenssh-gitshell-パッチ、nginxの、uwsgi、Redisは、memcachedの、beanstalkd、ジャンゴは、mysql、アーキテクチャが絶えず変化している使用してgitの、しかし、必ずしも最新の状態に保たれていない、私は今後の記事で詳しく説明しますアーキテクチャについて説明します。
gitshell限界？
ほとんどの開発者は通常使用することができますするために、技術的な要件と普通のウェブサイトより高いリソース消費をホスティングのgitを提供することにより、gitの操作上のいくつかの制限がありますが、現在gitshellは大規模なプロジェクトをサポートしていません、これはおそらく最も簡単な混乱していますすべてgitshellユーザーはこのことを理解する必要があります。

何ができるのか導入に加えて、それはについて話をすると行うことができないことが必要である。

良好なコミュニティの雰囲気、gitshell明確なルールを維持するために：

  • 質問は、グーグルのスキルの問題に注意を払う、"のように、上、招待コード、私にメールを送信"することができます聞かないでください...これらの言葉は表示できません。
  • 低炭素環境ではなく、ネットワークディスクとして、バックアップとして任意のファイルgitshellを入れてはいけません。ここでの唯一のコード関連の可能、バイナリファイルの多数を歓迎しない。印刷され、削除される可能性が高い文書の不適切なコンテンツがある場合。
  • 人、ない個人攻撃、汚い言葉、ロマンチックな、政治的トピック、テクスチャとの良好な、無意味な情報は、これらのトピックの議論の範囲内ではありません対応する場所を持っていなく、ここ、ここ、そして唯一の開発者に関連した。
  • 上記の規則に違反した場合、アカウントを無効にするのに十分深刻対応する制裁措置がありますが、私はこのような状況を持ってしたくない。

SSHキーの管理

SSH公開鍵のペアは、単に最初のセキュリティドアのように、すべての安全の基礎である。 ssh経由でデータを交換するための任意の合意は、SSHキー保護は良いプログラマであることをセキュリティ上の理由からgitshellは、安全意識を持っている必要があり、あなたはほかに、それは定期的にキーを更新するのが最善ですが、1Passwordの適切に保存を通じてツールのように願っています。

管理倉庫

倉庫、コンテナの基礎を開発することである、すべての活動は、倉庫から派生しています。あなたは人々があなたの倉庫をお知りになりたい場合は、それが最善の方法です
  1. グッドネーミング、シンプル輪郭。
  2. 人々はすべての問題を解決するためにあなたの倉庫に何をすべきか知っているように適切なコメントを、。
  3. もちろん、最も重要なのは、常に優れたアイテムになって開発しています。

コードへの倉庫のほかに、他のデータが厳しく制限されて、あなたはこれを理解する必要があります。

clone·push·pullコード

								
$ git clone git@gitshell.com:your/repo
$ cd repo; vim README.md
$ git add README.md
$ git commit -m "init README.md"
$ git push -u origin master
$ git pull

共同開発

gitshellは、エンタープライズレベルの開発とは異なる倉庫相乗的、の共同開発のメンバーの数を提供し、現在の権威は、それぞれの所有者の倉庫そう厳密に定義されていない最高の経営権、注意することは残りのメンバーと同等の権限があることのすべてのメンバー倉庫フルアクセス許可を読み書きします。グループとしての開発は、あなたが知っている必要があります：
  1. できるだけ他の人に迷惑を起こさない。
  2. 良いコミットレコードを持つ。
  3. コード構造は、清潔な澄んだ、との良好なコミュニケーションのすべてのメンバーに保つために。
